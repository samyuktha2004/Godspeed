# 01 ‚ÄĒ Query Routing Layer (Route-then-Retrieve)

> **Status: Implemented.** New deterministic `router_node` runs before the planner; a per-team "knowledge manifest" drives scope narrowing and agent pruning.

---

## The idea

Instead of letting the LLM planner fan out to every retrieval agent on every query, a cheap **deterministic router** runs first. It uses a per-team **knowledge manifest** (what sources/spaces/repos/projects exist) plus simple rules to:

- **narrow the search scope** (e.g. "this query is about the `ENG` Confluence space"), and
- **suggest which agents** to run (pruning fan-out),

then hands a `RoutingDecision` to the LLM planner, which still makes the final semantic choice. Determinism does the cheap, obvious narrowing; the LLM does what it's good at.

---

## Components

### Models ‚ÄĒ `agent/models.py`

```python
class RetrievalScope(BaseModel):
    source_types:  list[str] = []   # e.g. ["confluence"]
    space_keys:    list[str] = []   # Confluence space keys  (Qdrant payload: space_key)
    repos:         list[str] = []   # GitHub repos           (Qdrant payload: repo)
    jira_projects: list[str] = []   # agent-selection hint only (no Qdrant payload field)

class RoutingDecision(BaseModel):
    scope: RetrievalScope | None = None        # None ‚áí broad search (soft fallback)
    suggested_agents: list[str] = []
    confidence: Literal["high","medium","low"] = "low"
    reasoning: str = ""
```

`KnowledgeGraphState` gained a `routing_decision` field so the decision flows through the graph.

### Router ‚ÄĒ `agent/agents/router.py`

`run_router(query_input) -> RoutingDecision`. **No LLM call.** Signals, in order of strength:

| Signal | Example | Effect |
|---|---|---|
| Ticket-ID regex `\b[A-Z][A-Z0-9]+-\d+\b` | `KAN-7` | Jira scope + `ticket_lookup`, **high** confidence |
| Manifest entity match | query names a known space / repo / project | scope + relevant agent, **high** confidence |
| Aggregate keywords | "how many", "count", "stats" | suggests `sql_query` |
| Source keywords | "confluence", "slack", "jira/ticket" | suggests that agent, **medium** |
| Nothing matched | a vague question | **low**, no scope |

**Soft-routing policy (critical):** `scope` is populated **only when `confidence == "high"`**. For medium/low, `scope = None` and the search stays broad. `doc_search` is always kept as a fallback agent unless we are highly confident in a more specific source. This guarantees routing can never make a correct answer unreachable.

Manifest matching is conservative to avoid false narrowing: space keys match on whole-token equality, repo names require ‚Č•4-char token match, etc.

### Manifest read path

The router reads the manifest **Redis-first** (`gs:manifest:{team_id}`, 600 s TTL) and falls back to Supabase on a cache miss (lazy import of `manifest_store.get_routing_manifest`, run via `asyncio.to_thread`). On a Supabase hit it repopulates the cache. If no manifest exists yet, the router still works on ticket-id / keyword signals alone (graceful degradation).

`manifest_digest(manifest)` produces a one-paragraph summary that is fed to the planner prompt.

## The knowledge manifest

A compact per-team index of *what exists and where*, stored as `teams.routing_manifest` (jsonb). Shape:

```json
{
  "confluence": {"spaces":  {"ENG": {"doc_count": 42, "gist": "Engineering runbooks"}}},
  "github":     {"repos":   {"acme/api": {"doc_count": 30, "gist": "...", "top_paths": ["src"]}}},
  "jira":       {"projects":{"KAN": {"doc_count": 88, "gist": "..."}}},
  "file":       {"doc_count": 5},
  "pdf":        {"doc_count": 9},
  "updated_at": "..."
}
```

Two-speed freshness (`ingestion/storage/manifest_store.py`):

- **Structure (counts + which entities exist)** is recomputed from the `documents` table on **every ingest** via `refresh_manifest_structure()` ‚ÄĒ cheap, **no LLM, no re-embedding**. Existing gists are carried forward (`_merge_gists`).
- **Gists (a sentence per space/repo/project)** are regenerated by the **nightly CAG job** (`cag_job._refresh_team_manifest` ‚Üí `_generate_gists` via the existing Gemini helper).

Writers bust the Redis cache (`_bust_cache`) so updates show promptly. Jira project keys are derived defensively (`_jira_project`: explicit `project` field ‚Üí `issue_key` prefix ‚Üí configured default).

## Planner integration ‚ÄĒ `agent/agents/planner.py` + `agent/prompts.py`

`run_planner` now accepts `routing_decision` and `manifest_summary` and appends them to the Gemini user message. `PLANNER_SYSTEM_PROMPT` gained a rule: *prefer the suggested agents; at high confidence don't add others unless the query clearly spans more sources.* The `ExecutionPlan` output schema is **unchanged** ‚ÄĒ this only informs/prunes the selection.

---

## Scope as an additive Qdrant filter

When `scope` is present, `agent/tools/doc_search.py:build_scope_conditions` translates it into Qdrant `FieldCondition`s that are **ANDed onto** (never replace) the RBAC filter:

- `doc_search` applies `source_type`, `space_key`, `repo`.
- `confluence_search` applies `space_key` only (its `source_type` is already pinned).
- **Jira project** is *not* a Qdrant filter (no `project` payload field exists) ‚ÄĒ it's an agent-selection hint only.

Because scope is `None` unless the router is highly confident, behavior is identical to pre-change for any query the router can't place.

## Graph wiring ‚ÄĒ `agent/graph.py`

- `router_node` is the **entry point**, with a plain edge `router_node ‚Üí planner_node`.
- `router_node` emits a new SSE event **`routing_ready`** (before `plan_ready`), mirroring the existing `plan_ready` pattern, so the frontend can display the routing decision.
- Retrieval nodes read `state.routing_decision.scope` via `_current_scope(state)` and pass it to the tools.

**SSE sequence:** `routing_ready` ‚Üí `plan_ready` ‚Üí `agent_started`√óN ‚Üí `agent_done`√óN ‚Üí `synthesis_started` ‚Üí `answer_chunk`√ómany ‚Üí `citations` ‚Üí `guardrail_result` ‚Üí `done`.

## Verified behavior (isolated tests, real `run_router`)

- ticket-id ‚Üí high, jira scope, **only** `ticket_lookup` (fan-out pruned)
- known space/repo ‚Üí high, scoped, relevant agent
- vague query ‚Üí low, `scope=None`, broad `doc_search` fallback
- aggregate / slack keyword ‚Üí medium, `scope=None`, fallback kept
- works with no manifest present (ticket-id still routes)
